Definitions

Reference page. Every term in this document, one line each.

Clinical Event — One clinical fact extracted from a note. A stone on imaging, a drug given, a procedure done. The central object in this system.
Family — One of 5 clinical domains that group related event types: history_timeline, symptoms_imaging, procedure_devices, medications, outcomes_complications.
Event Type — One of 10 specific kinds of clinical facts within a family. Each family contains 1–3 event types. The event type determines which attributes that event carries.
Preferred Term — The canonical name for what an event describes. “ureteral stone”, “ureteroscopy”, “tamsulosin”. Normalized through Event Terms.
Concept ID — A stable identifier built from family + event_type + preferred_term. Example: symptoms_imaging.imaging_finding.ureteral_stone
Attributes — Key-value pairs that describe the details of an event. laterality=RIGHT, size=5, unit=mm. Each event type has its own attribute set.
Evidence Set — The exact quoted text from the note with character offsets and section name. Every event traces back to the sentence it came from.
Assertion Strength — How certain the clinician was. CONFIRMED = definitely present. POSSIBLE = suspected. NEGATED = explicitly ruled out.
Temporal Status — When relative to this visit. CURRENT = happening now. HISTORICAL = happened before. FUTURE = planned.
Experiencer — Who this happened to. PATIENT or FAMILY_MEMBER (for family history).
Event Terms — 990 aliases that map to 252 canonical preferred terms. Normalizes the name of the thing. “UVJ calculus” → “ureteral stone”.
Attribute Terms — 910 aliases that map to 327 canonical attribute values. Normalizes the details. “right” → RIGHT, “ureterovesicular junction” → UVJ.
Normalization Status — CANONICAL = alias found in vocab. PROVISIONAL = new term, flagged for review. CUSTOM = has a user-provided concept_id.
Attribute Policy — 89 rules defining what values are allowed per attribute per event type. Each rule specifies kind, style, and provisional.
attribute_kind — controlled = must be in vocab. semi_structured = follows a pattern (dates, measurements). free_text = any value allowed.
canonical_style — How the value is formatted. upper_tokens = RIGHT, LEFT. lower_text = nonobstructing. clinical_mixed = UVJ, UPJ. Others: measurement_unit, numeric_or_text, date_or_span, free_text.
allow_provisional — true = unrecognized values accepted and flagged. false = unrecognized values rejected outright.
Relation — A typed link between two events in the same note. 9 types: performed_for, caused_by, follow_up_for, treated_by, decompresses, temporally_linked, modifies, resolved_after, causes_obstruction.
Projection — The process of computing answers from events. Events go in, answers to clinician questions come out.
Resolver — The function that reads events and extracts the right data to answer one question. Each of the 130 questions has exactly one resolver.
event_presence — Checks if a matching event exists. Returns yes/no. Example: “Was a stent placed?”
event_attribute_values — Reads a specific attribute from matching events. Example: “Where is the stone?” → reads imaging_finding.location.
event_concepts — Matches events against keyword lists. Example: “Did symptoms resolve?” → matches outcome_event against [“symptom resolution”, “pain resolved”].
event_grouped_counts — Counts events grouped by a field. Example: “How many medications?” → counts medication_exposure events.
Registry Question — One of 130 clinician-designed questions. Each has a registry_id, a resolver, and a priority score from clinician voting.
registry_id — Unique identifier for each question. Example: stone_location, was_stent_placed.
Resolution State — Whether a question was answered or unanswered for a given note. Unanswered means the note had no events matching the resolver.
Answer Scope — What data the question looks at. note = this note only. patient_timeline = all notes for this patient. encounter = this visit. (expanded)

1. The Problem

A note states: "Punctate 2 mm nonobstructing calculus at the right ureterovesicular junction." The question is how do you take that and make it structured. How do you get from a sentence a surgeon dictated into a queryable fact with a laterality, a location, a size, and a canonical name that matches every other note that mentions the same kind of stone.

2. The Data Model: Why These Elements

The central object in this system is the clinical event. An event represents one clinical fact extracted from a note: a stone found on imaging, a drug administered, a procedure performed, a complication documented. The model was designed around one question: what are all the distinct things a clinician writes about in a kidney stone note that a researcher would want to query later?

Kidney stone care follows a natural clinical arc. Five families cover it. Each one maps to a concern urologists think about as a unit.

Family Clinical Concern Event Types What It Captures
history_timeline What happened before this visit? history_fact, social_context_fact Prior stones, family history, anatomic variants, caregiver/transportation barriers
symptoms_imaging What are we seeing now? symptom_presentation, imaging_finding Flank pain, hematuria, stone location/size/laterality, hydronephrosis, obstruction
procedure_devices What did we do? procedure_event, device_event Ureteroscopy, PCNL, stent placement, nephrostomy tubes, laser settings
medications What drugs were given? medication_exposure Drug name, dose, route, frequency, indication, context (ED, home, intra-op)
outcomes_complications What was the result? adverse_effect, outcome_event, recommendation_event Complications, stone-free status, symptom resolution, follow-up instructions
Fewer families means less ambiguity about where a concept belongs. Symptoms and imaging go together because clinicians evaluate them together.

Each event type captures a distinct clinical fact. Ten types across five families. A family is the clinical domain, the event type is the specific thing inside that domain. procedure_devices has two event types because the surgery itself and the stent left behind are different facts with different attributes and different research questions. The table below maps every type to its parent family.

Event TypeFamilyWhat It RepresentsExample

What Goes Inside an Event

Every event carries the same core fields. These were chosen so that each event is self-contained: you can look at a single event and know what it is, when it happened relative to this visit, how certain the clinician was, and exactly where in the note it came from.

FieldWhat It RecordsExample
event_idUnique ID within this noteEV1, EV2, EV3...
familyWhich of the 5 clinical domainssymptoms_imaging
event_typeWhich of the 10 typesimaging_finding
preferred_termCanonical name for the clinical conceptureteral stone
attributesKey-value pairs specific to the event type{laterality: RIGHT, size: 5, unit: mm}
temporal_statusWhen relative to this visitCURRENT, HISTORICAL, or FUTURE
assertion_strengthHow certain the clinician wasCONFIRMED, POSSIBLE, or NEGATED

evidence_setExact quoted text with character offsets and section"5mm calculus..." at chars 1204-1261
relationsLinks to other events in the same noteperformed_for -> EV1
"Kidney stone" means three different things depending on context. A CURRENT stone is why the patient is here today. A HISTORICAL stone means they are a recurrent stone former. A NEGATED stone ("no evidence of calculus") means imaging was performed and nothing was found. That is clinically different from never having been imaged. Without temporal_status and assertion_strength, a query cannot distinguish these cases.

How the Pieces Connect

Below is every step that turns a sentence into a queryable fact. One real sentence from an ED note, every transformation, real data at every tier.

Clinical Note is the raw input. A surgeon dictated "Punctate 2 mm nonobstructing calculus at the right ureterovesicular junction" and that sentence needs to become structured data. Event Terms normalize the name of the thing. The note says "UVJ calculus" but across 9,869 notes that same stone shows up as "ureterovesicular junction stone" and "distal ureteral calculus" so 990 aliases collapse to 252 canonical names and every spelling lands on one term. Attribute Terms normalize the details about that thing. "right" becomes RIGHT, "ureterovesicular junction" becomes UVJ. 910 aliases map to 327 canonical values. Assembled Event is what comes out. It has a family (the clinical domain like symptoms_imaging), an event_type (the specific thing inside that domain like imaging_finding), a preferred_term from Event Terms, and all the normalized attributes from Attribute Terms. Each family contains one to three event types and the event type determines which attributes that event carries. Attribute Policy is the rulebook. 89 rules say which values are allowed for each attribute on each event type. Take laterality on an imaging_finding: the policy says kind=controlled, style=upper_tokens, provisional=false. That means only LEFT, RIGHT, BILATERAL are legal. The model produces "right", normalization maps it to RIGHT, policy checks RIGHT against the vocab, it passes. If the model had produced "both sides" and normalization could not map it, provisional=false means reject it outright. That value never enters the dataset. Relations connect events within the same note. A medication was given for a stone, a stent was placed during a procedure. When one event is not enough to answer a question you follow the links. Projection is where events become answers. A resolver is just a function that reads events and pulls the right data out to answer a specific question. The clinician asks "Where is the stone?" The resolver for that question is event_attribute_values. It finds all imaging_finding events in the note, reads their location attribute, and returns UVJ. A different question like "Was a stent placed?" uses event_presence. It looks for device_event events with device_type=ureteral stent and returns yes or no. Four resolver types total: event_presence (does a thing exist), event_attribute_values (read a specific field), event_concepts (match events against keyword lists), event_grouped_counts (count across multiple events). Registry Questions are the 130 questions clinicians designed and prioritized. Each question has exactly one resolver wired to it and the resolver knows which event types to query and which attributes to read.

3. What Each Event Type Captures

Each event type has its own set of attributes. These are the specific details a clinician documents and a researcher needs. The attributes were chosen by reviewing the 130 registry questions and asking: what structured fields do we need in order to answer each question? For example, the question "What is the laterality of the stone?" requires that imaging_finding events carry a laterality attribute.

Imaging Finding

The most attribute-rich event type, because stone characterization drives treatment decisions. A 2mm nonobstructing stone is managed differently from a 2cm staghorn calculus.

AttributeWhat It RecordsExample Values
lateralityWhich sideLEFT, RIGHT, BILATERAL
locationAnatomic positionureter, kidney, UVJ, UPJ, renal pelvis, bladder

stone_burdenHow many or how muchsingle, multiple, staghorn, punctate
severityGrade if statedmild, moderate, severe (for hydronephrosis)
hounsfield_unitCT density800 HU

Symptom Presentation

Captures the symptom cluster the patient presented with. One event per symptomatic episode, not one event per symptom, because clinicians describe them as a group.

One event per distinct procedure. Operative details like laser settings and access approach are captured as attributes rather than separate events.

Devices are tracked separately from procedures because "was a stent left in place?" and "what procedure was done?" are different research questions.

Medication Exposure

One event per distinct drug. Medications relevant to stone care include pain control, alpha-blockers for medical expulsive therapy, antibiotics, antiemetics, and stone prevention drugs.

AttributeWhat It RecordsExample Values

Adverse Effect, Outcome Event, Recommendation Event

These three types cover the results of care. Adverse effects are complications (AKI, pyelonephritis, stent symptoms). Outcome events are treatment results (stone passage, stone-free status). Recommendation events are instructions for what comes next (follow-up, diet changes, strain urine).

Event TypeKey AttributesExample
adverse_effect reaction_type, severity, suspected_cause, timing, status AKI, severity=moderate, suspected_cause=obstruction
outcome_event status, value, qualifier, timing stone-free status, value=true
recommendation_event recommendation_type, target, status follow_up, target=urology, status=scheduled

History Fact and Social Context Fact

History facts capture what happened before this visit: prior stones, prior surgeries, family history of stones, anatomic variants like horseshoe kidney. Social context facts capture support barriers: does the patient have transportation? Is a caregiver available? These are tracked because they affect treatment decisions and discharge planning.

Event TypeKey AttributesExample
history_fact history_kind, subject, count, intervention_status kidney stone history, count=3, subject=PATIENT
social_context_fact support_type, value support_type=caregiver, value=available

5. From Events to Answers: The Registry

Events are the raw material. The registry is what makes them useful for research. 130 questions, designed by clinicians, imported from a prioritization workbook where each question was scored for clinical importance. The system computes answers automatically by looking at the events in each note.

What a Registry Question Looks Like

Each question has a unique ID, the original text as written by clinicians, the family it belongs to, and instructions for how to compute the answer from events. Here is the structure:

FieldWhat It StoresExample
registry_idUnique question identifierwhat_medication_was_given
clinician_textOriginal question as written"What medication was given?"
familyWhich clinical domain answers itmedications
answer_scopePer-note or across all notesnote (or patient_timeline)
answer_shapeType of answer returnedboolean, list, count, or grouped_count
resolver_idLogic for computing the answerevent_concepts, event_presence, etc.

Answer Shapes

There are four ways a question can be answered. The shape determines what the answer looks like and how the system computes it from the extracted events:

ShapeWhat It ReturnsExample QuestionExample Answer
boolean Yes or No "Was the patient symptomatic?" true
list One or more values "What medication was given?" ["tamsulosin", "morphine"]

grouped_count Values grouped with counts "Medications by class" [{"opioid": 2}, {"NSAID": 1}]

When the system computes an answer, it stores it as a "projection." Think of it as the receipt: it records which question was answered, what the answer is, and which events prove it.

6. Building Cohort Definitions

Because every note produces structured events and answered questions, you can query across all 764 annotated notes to define patient cohorts for research. A cohort definition is just a filter on the structured data.

Example Queries

Patients with obstructing ureteral stones who received tamsulosin

family = symptoms_imaging, event_type = imaging_finding, preferred_term = "ureteral stone", attributes.status = "obstructing"
AND
family = medications, event_type = medication_exposure, preferred_term = "tamsulosin"
JOIN on patient_id

This query works because the stone's status (obstructing vs. nonobstructing) is stored as a structured attribute on the imaging_finding event, and each medication is its own event with a canonical preferred_term. You join across families on patient_id.

Patients who had ureteroscopy and developed a post-operative complication

family = procedure_devices, event_type = procedure_event, preferred_term contains "ureteroscopy"
AND
family = outcomes_complications, event_type = adverse_effect
JOIN on patient_id

Stone-free rate after ureteroscopy

registry_id = "stone_free_status", resolution_state = "answered"
FOR patients with
event_type = procedure_event, preferred_term contains "ureteroscopy"

The third query demonstrates why the projection layer matters. Instead of scanning all events to infer "stone-free," you query the pre-computed answer to the "stone-free status" question directly. The projection already did the work of checking whether the right outcome_event exists for each note.

7. Connecting Events: Relations and Context Labels

Relations Between Events

Events within the same note can be linked to each other. A procedure can be linked to the stone it was treating. A complication can be linked to the procedure that caused it. A follow-up recommendation can be linked to the condition it is monitoring. These links let researchers ask questions like "what complications occurred after ureteroscopy?" by following the relation edges.

Relation TypeMeaningExample
performed_forProcedure done to treat a findingureteroscopy performed_for ureteral stone
caused_byComplication caused by a treatmentAKI caused_by obstruction
follow_up_forRecommendation monitors a conditionurology follow-up follow_up_for kidney stone
treated_byFinding treated by a procedurehydronephrosis treated_by stent placement
decompressesDevice relieves obstructionnephrostomy tube decompresses hydronephrosis
temporally_linkedEvents happened in sequencestent placed temporally_linked to ureteroscopy
modifiesOne event refines anotherstone analysis modifies imaging finding
resolved_afterCondition resolved after treatmenthydronephrosis resolved_after stent placement
causes_obstructionStone causes urinary obstructionureteral stone causes_obstruction

Each relation records which two events are connected, the type of connection, and a brief rationale explaining why the link exists. Relations are stored as part of the source event and point to the target event by ID.

Temporal Status

Every event is tagged with when it happened relative to the current visit. This is critical for answering time-sensitive questions correctly:

ValueMeaningNote Text Example
CURRENT This visit or actively present now "5mm stone in right ureter"
HISTORICAL Happened before this visit "history of nephrolithiasis"
FUTURE Planned but not yet done "will undergo ureteroscopy next week"
A HISTORICAL procedure does not answer "what procedure was performed this visit?" A FUTURE recommendation is not a completed one. Without temporal tagging, a query for "patients who had ureteroscopy" would incorrectly include patients who only had it planned or who had it years ago.

Assertion Strength

Clinicians do not always state things with certainty. The system captures three levels:

ValueMeaningNote Text Example
CONFIRMED Definitively present or performed "CT shows 5mm calculus"
POSSIBLE Uncertain, pending confirmation "possible robotic pyeloplasty pending results"
NEGATED Explicitly ruled out by the clinician "without definite finding of ureteral calculus"

NEGATED events are extracted and stored, but they do not produce positive answers to clinical questions. This preserves the clinically important distinction between "we looked and it was not there" and "we did not look."

Experiencer

Most events happen to the patient, but some notes mention family history of kidney stones. The experiencer field tracks who:

ValueWhen Used
PATIENTThe event happened to the patient (default, vast majority of events)
FAMILY_MEMBERFamily history: "mother has kidney stones"

Projection Exclusion Rules

Not every extracted event contributes a positive answer to a question. The projection system applies exclusion rules so that only clinically appropriate events produce results:

ConditionEffect on Projection
NEGATED assertionExcluded from positive answers ("stone not seen" does not answer "Where is the stone?")
FUTURE temporal statusExcluded. Planned events have not occurred yet.
HISTORICAL temporal statusExcluded from note-scope questions (but included in timeline-scope)
PROVISIONAL / CUSTOM eventsExcluded. Only fully validated events project.
Procedure status: planned / scheduled / deferred / canceledExcluded from procedure-completion questions
Without exclusion rules, a query for "was ureteroscopy performed?" would count future planned procedures and negated findings as positive answers. The rules ensure projections reflect what actually happened to the patient during the encounter.

Each annotated note produces a JSON artifact containing the full extraction:

{ "schema_version": "event-first", "patient_id": "HID.907...", "note_id": "OUT.IDX.0004721", "events": [ { "event_id": "evt-...", "event_type": "imaging_finding", "preferred_term": "hydronephrosis", "assertion_strength": "CONFIRMED", "temporal_status": "CURRENT", "attributes": { "laterality": "RIGHT", "severity": "moderate" }, "evidence": { "anchors": [{ "quote": "Right renal hydronephrosis", ... }] }, "field_provenance": { "preferred_term": { "source": "model", "overwrote_model": false }, "assertion_strength": { "source": "model", "overwrote_model": false }, "laterality": { "source": "candidate_binding", "overwrote_model": false } }, "relations": [...] } ], "projections": [ { "registry_id": "Q041", "answer": ["RIGHT"], "resolver": "event_attribute_values" } ] }

9. Controlled Vocabularies

Raw clinical text is inconsistent. A surgeon writes "URS," another writes "ureteroscopy," a third writes "flexible ureteroscopy." All three mean the same procedure. The system handles this through two parallel vocabulary layers: one for event preferred terms, one for attribute values.

How Normalization Works

When the extraction model proposes an event, the normalization engine receives the raw preferred_term and each attribute value. It lowercases and strips punctuation to build a lookup key, then checks that key against the vocabulary tables. If a match exists, the raw text is replaced with the canonical form.

Model output: "rt hydroureteronephrosis"
↓
Lookup key: "rt hydroureteronephrosis"
↓
Canonical term: "hydroureteronephrosis"
↓
Status: CANONICAL (laterality extracted to attribute)

If no alias matches, the event gets a PROVISIONAL status and a generated concept_id. PROVISIONAL events are valid extractions. They just indicate the vocabulary has not seen this exact phrasing before. The gap-reporting system flags these so new aliases can be added over time.

Event Preferred Terms

The event vocabulary maps 990 aliases to 252 canonical preferred terms across all 10 event types. Each (family, event_type) pair has its own lookup table. A few examples from the imaging_finding vocabulary:

"ureteral calculus" → "ureteral stone" "uvj stone" → "ureteral stone" "distal ureteral stone" → "ureteral stone" "obstructing ureteral calculus" → "ureteral stone" "mild right hydronephrosis" → "hydronephrosis" "moderate left hydronephrosis" → "hydronephrosis" "staghorn calculi" → "staghorn calculus"

The canonical term ("ureteral stone") is what gets stored in the event. Laterality, severity, and location are parsed into separate attributes. This separation prevents a combinatorial explosion. Instead of maintaining entries for every combination of laterality + severity + finding, the system normalizes the finding once and captures the modifiers as structured attribute values.

Attribute Vocabularies

The attribute vocabulary covers 57 controlled attributes across all event types, mapping 910 aliases to 327 canonical values. Each attribute is scoped to a specific (family, event_type, attribute) triple. Here is how the symptom vocabulary works:

"left flank pain" → "LEFT FLANK PAIN" (upper_tokens style) "l flank pain" → "LEFT FLANK PAIN" "cva tenderness" → "CVA TENDERNESS" "costovertebral angle tenderness" → "CVA TENDERNESS" "gross hematuria" → "HEMATURIA" "blood in urine" → "HEMATURIA" "colicky flank pain" → "RENAL COLIC"

Canonical style varies by attribute. Symptoms use UPPER_TOKENS because they appear in structured output tables alongside coded values. Medication routes use the same convention (IV, PO, IM). Drug classes use lower_text (opioid, NSAID, alpha-blocker) because they read more naturally in prose summaries.

Vocab Guidelines: The Attribute Policy Layer

Each of the 57 attributes has an explicit policy that defines three things: what kind of vocabulary it uses, how canonical values are formatted, and whether the extraction model is allowed to introduce new values.

PropertyOptionsWhat It Controls
attribute_kindcontrolled, semi_structured, free_textWhether values must come from the vocabulary (controlled), follow a pattern like dates or measurements (semi_structured), or allow any text (free_text)
canonical_styleupper_tokens, lower_text, lower_or_snake, clinical_mixed, measurement_unit, numeric_or_text, date_or_span, free_textThe formatting convention for canonical values
allow_provisionaltrue / falseWhether unrecognized values are accepted (true) or rejected (false). Controlled attributes with allow_provisional=false will reject any value not in the vocabulary.

A concrete example. The attribute imaging_finding.laterality has kind=controlled, style=upper_tokens, allow_provisional=false. That means the only legal values are LEFT, RIGHT, and BILATERAL. If the model produces "rt" or "bilateral kidneys," the normalization engine maps it to the canonical form. If it produces something unmappable, the value is rejected and flagged.

Contrast that with procedure_event.indication: kind=controlled, style=lower_text, allow_provisional=true. The vocabulary has 9 canonical values (stone removal, stent placement, drainage, etc.), but if the model encounters a genuinely new indication like "stent exchange," it can accept the provisional value while the gap reporter flags it for vocabulary expansion.

The vocabulary system trades upfront curation effort for downstream consistency. When a researcher queries "all patients with hydronephrosis," they get every instance. Not just the ones where the surgeon happened to write "hydronephrosis" rather than "mild right hydroureteronephrosis." The 910 attribute aliases and 990 event term aliases are what make that guarantee possible.
